# M1：存量资源集中迁移设计

## 背景

阶段 7A 到 7D 已经完成了存量目录盘点、资源配对、安全执行计划、复制整理和低空间硬链接整理。它们证明了“从多个真实目录扫描并按番号聚合资源”这条路线是有效的，但 7C/7D 的执行策略仍然偏保守：

- 7C 只复制，不删除源文件。
- 7D 视频硬链接、小资源复制，也不删除源文件。

用户当前的核心需求不是保留源目录，而是把散落在不同目录里的视频、NFO、图片、GIF 等资源，按既定的每番号目录格式集中到一个目标位置，并让成功迁移后的源文件不再留在原位置。M1 因此从“低风险整理预演”切换为“真实集中迁移”。

## 目标

- 在“一键盘点”的安全执行计划基础上新增集中迁移执行模式，默认面向真实存量整理。
- 成功迁移后，源文件必须消失，目标目录必须保留校验后的文件。
- 同一卷/同一盘符内优先使用直接移动，不额外复制大视频内容。
- 跨卷/跨盘符时按单文件顺序执行：复制到目标临时文件、校验、提交目标、删除源文件。
- 每个跨卷文件迁移前检查目标卷剩余空间；空间不足时停止，不删除源文件。
- 不覆盖任何已存在目标文件；目标冲突必须在执行前或执行时失败。
- 执行过程有明确 loading、确认文案、完成摘要和逐文件日志。
- 失败时保留可审计报告，让用户知道哪些文件已经迁移、哪些文件未动、失败原因是什么。
- 复用 7B.1 之后的 `resolution.execution_plan.actions`，不直接消费早期候选动作。

## 非目标

- 不自动执行 `needs_review`、`blocked`、`asset_candidate` 或孤儿资源。
- 不用远程 scraper 猜测孤儿资源番号。
- 不在 M1 中做 SQLite 入库同步、播放库重建或封面缓存刷新。
- 不删除非空源目录。
- 不做跨设备批量并发搬运；M1 必须保持单文件顺序，避免瞬时占用过多磁盘空间。
- 不继续扩展旧的“集中迁移 / 智能迁移”实验入口作为主线功能。

## 关键定义

- 集中迁移：把源文件搬到目标路径。成功后目标文件存在，源文件不存在。
- 同卷移动：源文件和目标目录位于同一文件系统/卷时，使用重命名式移动。
- 跨卷迁移：无法直接重命名时，使用复制、校验、提交、删除源文件的安全序列。
- 安全执行计划：`InventoryWorkPreview.resolution.execution_plan.actions`。只有 `ready == true` 且无冲突的计划可以进入执行。
- 作品级提交：一个作品内的文件全部成功后，该作品视为迁移完成；后续作品失败不会回滚已经完成的作品。
- 作品级回滚：同一个作品处理中途失败时，执行器尽量把本作品已经迁出的文件移回原路径或删除未提交目标。

## 目标目录规范

M1 继续沿用 7A 已定的目标布局：

```text
archive_root/
  CODE/
    CODE.ext
    CODE.nfo
    poster.ext
    fanart.ext
    thumb.ext
    screenshots/
      原文件名.ext
    gifs/
      原文件名.gif
    images/
      原文件名.ext
```

其中：

- 视频：单视频使用 `CODE.ext`；多版本继续使用安全执行计划生成的目标名。
- NFO：使用 `CODE.nfo`。
- poster/fanart/thumb：保留原扩展名，目标 stem 规范化为角色名。
- screenshot/GIF/generic image：保留原文件名，放入对应子目录。
- other：只在安全执行计划明确选入时迁移，目标路径沿用计划。

## 执行入口

新增执行模式：

- Rust：`InventoryExecutionMode::Move`
- JSON / TypeScript：`"move"`

前端“一键盘点”面板的主执行按钮从“低空间整理”升级为“集中迁移”。按钮只在以下条件同时满足时可用：

- 已有盘点报告。
- 后端可用。
- 存在 `auto_ready` 且安全执行计划 `ready == true` 的作品。
- 作品明细未被截断，即 `summary.works <= works.length`。
- 当前没有盘点、导出或执行操作。

执行前确认框必须明确说明：

- 本操作会移动文件，成功后源路径不再保留。
- 同盘直接移动，跨盘会逐文件复制校验后删除源文件。
- 目标已存在不会覆盖。
- 失败会停止后续队列，并显示失败文件和原因。

## 文件操作策略

执行器先复用 7C/7D 的预检：

- 源文件必须存在且是普通文件。
- 目标路径必须位于 `archive_root` 下。
- 目标父目录解析 symlink/junction 后仍在 `archive_root` 下。
- 目标文件当前不存在。
- 同批执行目标路径不重复。
- 只执行 `auto_ready` 作品的安全执行计划。

随后按作品、按动作顺序执行。

### 同卷移动

同卷移动用于源文件和目标父目录位于同一卷的情况：

1. 创建目标父目录。
2. 再次确认目标不存在。
3. 执行 no-clobber 移动。
4. 校验目标文件大小与源文件迁移前大小一致。
5. 确认源路径已经不存在。
6. 记录 `moved` 日志。

同卷移动不需要额外目标盘空间，但仍需要目标目录元数据写入权限。

### 跨卷迁移

跨卷迁移用于直接移动不可用或源/目标位于不同卷的情况：

1. 读取源文件大小。
2. 查询目标卷剩余空间。
3. 若剩余空间小于 `文件大小 + 安全余量`，停止执行，源文件不动。
4. 在目标父目录创建唯一临时文件。
5. 流式复制源文件到临时文件，复制过程中计算源 SHA-256，并同步落盘。
6. 校验临时文件大小和 SHA-256。
7. 再次确认最终目标不存在。
8. 将临时文件提交为最终目标。
9. 再次校验最终目标大小和 SHA-256。
10. 删除源文件。
11. 确认源路径不存在。
12. 记录 `moved` 日志，日志消息标明 `copy_verify_delete`。

安全余量第一版固定为 512 MiB。后续如果真实环境证明不够，再做可配置项。

### 回滚边界

M1 不做全局大事务，避免把已经成功迁移的大视频再搬回去。回滚边界限定在当前作品：

- 当前作品某个动作失败时，停止该作品和后续队列。
- 已经移动到目标的本作品文件，若目标仍与本轮记录的校验信息匹配，尝试迁回原路径。
- 同卷回滚优先直接移动回原路径。
- 跨卷回滚按复制校验删除目标执行，但只在源路径不存在且空间允许时尝试。
- 回滚失败必须记录 `rollback_failed`，不继续处理后续文件。
- 已经完成提交的前序作品不回滚，报告里标记为已完成。

这个边界牺牲了全局原子性，但符合真实媒体库的成本约束：一次只处理一个文件，不会因为后续失败反复复制大量视频。

## 报告模型

继续复用并扩展 `InventoryExecutionReport`：

- `mode`: 新增 `move`。
- `moved_actions`: 成功迁移动作数。
- `bytes_moved`: 成功迁移的源文件总字节数。
- `same_volume_actions`: 同卷直接移动动作数。
- `cross_volume_actions`: 跨卷复制校验删除动作数。
- `space_blocked_actions`: 因目标空间不足停止的动作数。
- `logs.status`: 新增 `moved`、`rollback_failed`。
- `logs.message`: 记录 `rename`、`copy_verify_delete`、空间不足、目标冲突、回滚结果等细节。

原有 `copy` 和 `low_space` 模式继续保留，避免破坏既有测试和诊断能力；但 UI 主入口切到 `move`。

## UI 设计

“一键盘点”面板保留盘点和筛选能力，执行区调整为：

- 主按钮：`集中迁移`
- 次要按钮：保留 `导出 JSON`
- 状态行：`正在集中迁移安全执行计划...`
- 完成摘要：`集中迁移完成：作品 A/B，迁移文件 X，失败 Y，回滚 Z，同盘移动 M，跨盘迁移 N。`
- 最近执行日志：展示状态、资源类型、大小、源路径、目标路径和失败原因。

执行期间：

- 禁用入口目录、整理目标目录、盘点、导出和执行按钮。
- 显示当前操作是同盘移动还是跨盘复制校验删除。
- 失败后保留最近执行日志，不自动清空盘点结果。

旧“统一迁移/智能迁移”区域在 M1 后不再作为推荐路径。实现阶段可以先加“旧入口”标识，M1 验证稳定后再移除或折叠，避免用户继续误用。

## 磁盘空间策略

M1 只对跨卷迁移检查目标卷空间，因为同卷移动不复制文件内容。

空间检查需要抽象成可测试接口：

- 生产环境：读取目标父目录所在卷的可用空间。
- 单元测试：用 fake provider 模拟足够空间、空间不足、查询失败。

查询失败时不应盲目迁移大文件。M1 默认把空间查询失败当成阻断，提示用户检查目标盘状态。

## 安全边界

- 目标路径存在时永不覆盖。
- 源文件删除只发生在目标文件已经校验成功之后。
- 临时文件只创建在目标父目录下，避免跨目录提交。
- 临时文件名包含 run id 和动作序号，避免与用户文件冲突。
- 回滚只处理本轮日志记录过的路径，不扫描或猜测外部文件。
- 同卷移动不强制为大视频计算全文件 SHA-256；它没有数据复制链路，默认用大小、源路径消失和目标路径存在做校验，避免把同盘迁移拖成全盘读取。
- 跨卷迁移必须使用大小和 SHA-256 校验，因为它经过真实复制链路。
- 不跟随待扫描目录中的 symlink；目标父目录继续做 canonical 校验。
- 不通过 PowerShell 修改含中文的前端源文件。

## 测试策略

Rust 测试优先覆盖核心执行器，不依赖真实 H/G 盘：

- `move` 模式同卷迁移：源文件消失，目标文件存在，内容一致，报告为 `moved`。
- 目标已存在：执行前失败，源文件不动，目标不被覆盖。
- 跨卷策略：用 fake mover / fake volume detector 触发 `copy_verify_delete` 路径，不要求真实跨盘。
- 空间不足：报告 `space_blocked_actions == 1`，源文件不动，目标不存在。
- 复制后校验失败：源文件不动，临时文件清理。
- 当前作品中途失败：本作品已迁动作尽量回滚，前序作品保持完成。
- 回滚失败：报告 `rollback_failed`，停止后续动作。
- 非 `auto_ready`、作品明细截断、目标越界、重复目标等 7C 安全边界继续有效。

前端测试覆盖：

- 点击“集中迁移”调用 `executeInventoryPlan(report, [], "move")`。
- 确认框明确出现“源路径不再保留”和“跨盘复制校验后删除源文件”。
- 执行中按钮 disabled 且文案切换。
- `formatInventoryExecutionSummary` 正确显示同盘、跨盘、迁移、失败和回滚统计。

完整验证仍使用：

```powershell
cargo test --manifest-path src-tauri/Cargo.toml -j 1
npm test
npx tsc --noEmit
npm run build
```

不得在 Codex 会话中启动 `tauri dev`、`cargo run` 或 `media-manager.exe`。

## 交付拆分

M1 建议分三步交付：

1. **M1A 设计与计划**：固化本设计，写 TDD 实施计划。
2. **M1B 后端执行器**：实现 `move` 模式、空间检查、同卷移动、跨卷复制校验删除、报告和 Rust 测试。
3. **M1C 前端入口**：把“一键盘点”主按钮切到集中迁移，补确认文案、执行摘要和前端测试。

M1 完成后再进入后续阶段：迁移结果入库同步、目标库重建、人工复核项执行，以及旧迁移入口清理。
