# 媒体管理器重构设计

- 日期：2026-06-25
- 状态：待评审

- 关联 mockup：`.superpowers/brainstorm/1673-1782312494/content/` 下的各页面 HTML

---

## 1. 背景与重构动机

现有 `media-manager` 是一个 Tauri + React + Rust + SQLite 的本地媒体库工作台，从 Phase A 到 H 跑完了 ingest 扫描 / 识别 / 审核 / 归档流水线，并叠加了 NFO 元数据重建、作品库浏览、外部图片库绑定、资源池、迁移、增量同步等功能。

它已变成"负资产"，根因有三：

1. **定位漂移**：起步是"归档整理工作台"，后来一路往"媒体库浏览"加东西，又塞进"资源池 / 迁移 / 同步"，没有一个清晰稳定的产品核心。
2. **改动难以完成**：核心模型是"扫描 → 把没把握的全丢进 review 队列让人工逐条确认 / 合并 / 改码"，代码在打满补丁的旧架构上不断叠加，新需求牵动太多耦合旧逻辑。
3. **交互不符需求**：前端 `App.tsx` 是超大单体，画出来的界面和真实使用诉求对不上。

本次重构的目标：**甩掉旧架构包袱，复用已验证的核心，重建一个"端到端、尽量零干预"的自动管线 + 可检索浏览的媒体管理器。**

## 2. 目标与范围

### 2.1 产品目标

管理从**下载 → 自动归档 → 整理 → 分类移动 → 浏览播放**的整套流程，尽量不让用户手动整理数据，要看的时候又能方便找到。

### 2.2 本期范围（In Scope）

- 带标准番号（`LETTERS-DIGITS` 模式，日本厂商）的长片。
- 视频 + 同番号多版本 + 外部图片库（封面 / 截图 / GIF）的自动识别、整理、归档、浏览、播放。
- 深色沉浸式前端，覆盖六大页面（库浏览 / 详情 / 异常队列 / 搁置区 / 收藏夹 / 设置）。
- 开机自启的后台服务，监控目录、自动入库。

### 2.3 非目标（Out of Scope，本期搁置）

- 短视频、非日厂作品（识别不到标准番号的，进搁置区，不强行归档）。
- 内置 BT / 网盘下载器（只对接 aria2 的完成信号，不做下载本身）。
- 感知 / 近似去重（仅保留采样指纹的精确去重）。
- 多用户、远程访问、移动端。
- 内置视频播放器的深度功能（本期内置基础播放 + 续播进度，不做音轨/字幕/转码等高级能力）。

## 3. 核心决策摘要

| 维度 | 决策 |
|---|---|
| 定位 | 端到端自动管线 + 可检索浏览 |
| 自动化程度 | 默认全流程自动（含移动）；仅"番号冲突 / 疑似重复 / 刮削失败"三类异常才找用户 |
| 入库触发 | 后台服务监控目录 |
| 服务强度 | 开机自启的独立守护进程（托盘 daemon） |
| 架构 | 后台服务（Rust daemon）+ 前端（Tauri）+ SQLite，三层 |
| 范围 | 仅标准番号长片；其余进搁置区 |
| 元数据 | 联网刮削补全（番号 → 在线库） |
| 视觉基调 | 深色沉浸（Jellyfin 式），cyan `#00a4dc` 强调 |
| 播放 | 内置播放器（记录进度、支持续播） |

## 4. 整体架构

三层结构，职责分明，SQLite 为唯一共享真相源。

### 4.1 后台服务（daemon）

一个独立的 Rust 守护进程，登录后由托盘自启（用户态，不需 Windows Service / 管理员）。它承担所有"重活"：

- 监控资源目录（文件系统监听 `notify` + 启动 / 定时全量扫描兜底）。
- 跑完整入库管线：识别番号 → 联网刮削 → 整理 → 移动到归档目录 → 写 SQLite。
- 对接 aria2 RPC 判定"下载完成"。
- 开放一个**本地控制接口**（`127.0.0.1` HTTP / WebSocket），前端通过它触发操作、看实时进度、改配置。
- 常驻系统托盘，显示"监控中 / 处理中 / 已暂停"，可手动暂停 / 恢复。

### 4.2 前端（Tauri webview）

独立窗口，只做"看和点"：

- 浏览 / 搜索 / 筛选 / 播放，配置目录与刮削源，看异常队列与搁置区。
- 读 SQLite + 调服务的控制接口拿进度。
- **关掉它完全不影响入库**；重新打开接着用。
- 内置基础视频播放器（记录进度、续播）。

### 4.3 SQLite

服务与前端**唯一共享真相源**。服务写为主（管线产出），前端读为主（配置可写）。两端不抢写。

### 4.4 解决的旧坑

现有 `AGENTS.md` / `lessons.md` 反复记录的"无人值守启动 Tauri GUI / WebView2 就崩溃"——核心逻辑挪进独立服务进程、不再嵌在 GUI 里，该坑天然绕开。前端仅作为用户主动打开的窗口。

## 5. 自动管线（数据流）

文件从"下载完成"到"入库可搜"，默认全自动跑完六步：

### ① 监控（完成判定）

- aria2 RPC 轮询为主：`aria2.tellStatus`，`status=complete` 且 `completedLength==totalLength` 时判定完成（能正确处理 BT 多文件）。
- 完成回调（`--on-bt-download-complete`）做即时触发。
- 启发式兜底（aria2 未配置 / 非 aria2 来源）：`.aria2` 控制文件消失 + 文件大小稳定 + 可独占打开。
- 启动时 + 定时全量扫描兜底，保证关机期间攒下的文件也能补上。

### ② 识别

- 从文件名 / 路径提取标准番号（复用现有 `identifier`）。
- 提取到 → 继续。
- 提取不到 / 短视频 / 非日厂命名 → 进**搁置区**（不进异常队列打扰用户，单独放，可手动接管）。

### ③ 刮削

- 用番号查在线元数据库，补全 标题 / 演员 / 厂牌 / 系列 / 标签 / 封面 / 发行日。
- 多源组合，默认 FANZA/DMM 为主，回退 JavBus / JavDB；可配置、可拖拽排序、可单独开关。
- 成功 → 继续；失败（查无此号 / 网络错 / 源全挂）→ 进异常队列。

### ④ 整理

- 同番号多版本合并（`-cd2` / `-v2` 折叠为一个作品的多个文件版本，复用现有逻辑）。
- 按番号从图片库关联封面 / 截图 / GIF。

### ⑤ 移动

整理进归档目录，纯番号平铺：

```
<归档根>/<番号>/
    <番号>.nfo          # 刮削来的元数据，写成标准 NFO（归档目录自包含）
    <番号>.mp4          # 主视频（move）
    <番号>-v2.mp4       # 多版本
    poster.jpg / fanart.jpg / thumb.jpg
    screenshot/ *.jpg
    <番号>.gif
```

- 同番号已存在 → 合并进去，**不覆盖**已有文件。
- 移动是 move，失败则回滚、原文件不动，并进异常队列。

### ⑥ 入库

写 SQLite（works + 演员 / 标签 / 系列 / 厂牌关系 + 文件版本 + 刮削来源）。前端立刻能搜到。

### 异常边界（仅三类进异常队列）

1. **番号冲突**（文件名番号 ≠ NFO / 已有记录番号）
2. **疑似重复**（采样指纹命中库里已有作品）
3. **刮削失败**（所有刮削源都拿不到元数据）

其余全部自动过。

## 6. 数据模型（SQLite）

### 6.1 复用（清理后保留）

- `works` — 作品主表（番号、code_kind、标题 / 原名、厂牌、系列、发行日、评分、封面 / 剧照 / 缩略 / GIF 路径）。现有 32 列模型。
- 关系表：`work_actors` / `work_tags` / `work_sets` / `work_ratings`。
- `file_versions` — 文件版本（归档路径、原名、大小、时长、分辨率、编码、指纹 hash、归档状态）。
- `app_settings` — 配置（资源目录、归档根、刮削源开关、aria2 RPC 等）。

### 6.2 用户新增字段（挂在 works）

- `user_rating` — 用户自评分（0–10，独立于刮削评分）。
- `watch_status` — 未标记 / 想看 / 在看 / 看过 / 搁置。
- `watch_progress` — 上次播放位置（秒 + 百分比，用于续播）。
- `last_played_at` — 上次播放时间。

### 6.3 新增表

- `scrape_jobs` — 刮削记录（作品、源、状态、尝试次数、上次时间、错误），用于失败重试与追溯。
- `exceptions` — 异常队列（对象、类型、证据、状态、时间）。
- `holding` — 搁置区（无番号 / 短视频 / 非日厂文件、原因、可手动接管）。
- `pipeline_runs` — 管线执行日志（每次接管的文件、各步骤结果、耗时），用于排查与状态栏。
- `collections` — 收藏夹（id、名称、颜色标记、排序）。
- `work_collections` — 作品 ↔ 收藏夹多对多（一个作品可属于多个夹）。

## 7. 前端设计

### 7.1 视觉基调

深色沉浸（Jellyfin 式）：背景 `#0c0c11`，卡片 `rgba(255,255,255,.035)`，主文 `#f5f5f7`，次文 `#8a8a93`，强调 cyan `#00a4dc`。大圆角、柔和阴影、毛玻璃半透明、留白克制、SVG 线性图标（SF Symbols 风，1.7 描边）。macOS 窗口感（交通灯）。

### 7.2 六大页面

| 页面 | 形态 | 定稿 mockup |
|---|---|---|
| 媒体库 | 统一 2:3 卡片墙 + 搜索 + 常用标签（自定义）+ 状态角标 + 分页 | `library-dark.html` |
| 详情 | 沉浸式 backdrop + 海报 + 元数据 + 演员 + 动图(GIF 自动循环) + 相似推荐 + 续播进度 | `detail-recap.html` |
| 异常队列 | 三类异常卡片 + 一键处理 + 多选批量 + 右侧勾选 + 分页 | `exceptions-v3.html` |
| 搁置区 | 文件列表（无封面）+ 原因分类 + 指定番号 | `holding.html` |
| 收藏夹 | 夹子网格（4 封面拼贴）+ 点进卡片墙 | `collections.html` |
| 设置 | 资源目录(文件选择器) + 刮削源 + aria2 + 服务 + 缓存，紧凑表单 | `settings-v3.html` |

最近播放 = 复用媒体库卡片墙，按播放时间排序。

### 7.3 交互规范

- **弹窗与菜单**（`modals-v2.html`）：遮罩 `rgba(0,0,0,.65)` + 毛玻璃；卡片 `#1a1a22` 圆角 14；主按钮 cyan、次按钮半透明；`✕` / `ESC` / 点遮罩关闭；提示语单行不换行。已覆盖：新建/编辑收藏夹、加入收藏夹、手动指定番号、⋯ 更多菜单。
- **其余同类弹窗**（按同一规范，实现时落地）：筛选抽屉（库浏览"筛选"→ 演员/厂牌/系列/标签多选）、去搜索结果（番号冲突"去搜索"→ 搜索 + 结果列表）、图片放大 lightbox、内置播放器（全屏 + 进度/续播）、打开两个文件对比（疑似重复并排）、删除确认、常用标签添加。
- **文件 / 文件夹输入一律用系统选择器**（`@tauri-apps/plugin-dialog`），不手打路径。
- **操作反馈**：每个耗时操作（扫描、刮削、移动、重试）必须有显式 loading + 状态行（沿用现有 UX 原则）。

## 8. 异常处理

异常队列是用户唯一需要操心的地方，默认仅收三类：

| 类型 | 证据展示 | 一键处理动作 |
|---|---|---|
| 番号冲突 | 文件名番号 vs NFO 番号 并排 | 用文件名番号 / 用 NFO 番号 / 手动输入 / 两个都不对去搜索 |
| 疑似重复 | 新文件 ↔ 已入库作品，指纹相似度 | 链接为版本 / 强制新建 / 打开两文件对比 / 删除重复 |
| 刮削失败 | 各源失败状态（查无 / 超时） | 换源重试 / 手动填元数据 / 先入库稍后刮 / 忽略 |

- 支持多选批量：右侧勾选框 + sticky 批量栏；混选只给通用动作（忽略 / 删除），同类选中给专属批量动作。
- 分页。

## 9. 刮削器

- 可配置多源，默认顺序 FANZA/DMM → JavBus → JavDB；每源可单独开关、可拖拽排序。
- 番号 → 在线库查询，取回结构化元数据（标题 / 原名 / 演员 / 厂牌 / 系列 / 标签 / 封面 / 发行日 / 评分）。
- 单源失败自动回退下一源；全源失败 → 异常队列。
- 刮削结果写 `scrape_jobs` 记录来源与时间，便于追溯与重试。
- 用户编辑过的元数据字段为权威，后续刮削不覆盖（沿用现有"用户字段保护"规则）。

## 10. 服务部署与运行时

- **托盘 daemon**：独立 Rust 进程，登录后自启（启动项，非 Windows Service），托盘图标显示状态、可暂停/恢复/立即扫描。
- **本地控制接口**：daemon 监听 `127.0.0.1:<port>`，提供 REST（触发扫描/重试/配置）+ WebSocket（推送管线实时进度、异常计数）。前端调它。
- **aria2 集成**：通过 aria2 JSON-RPC（`aria2.tellActive` / `tellStatus`）轮询任务状态；可选 `--on-bt-download-complete` 回调做即时触发。
- **配置持久化**：资源目录、归档根、刮削源、aria2 RPC、自启开关等存 `app_settings`，daemon 启动时加载。
- **安全**：控制接口仅绑定 loopback，需会话密钥鉴权（类似本次 brainstorming 服务器的做法），防止本机其它进程 / 局域网误触。

## 11. 测试策略

- **Rust 核心**：`cargo test`，保留并迁移现有单元测试（identifier / scanner / nfo / storage / resource_pool / archive planner），新增刮削器、aria2 完成判定、异常判定的测试。
- **管线集成测试**：端到端 预置文件 → 监控触发 → 识别 → 整理 → 移动 → 入库 → 可搜，断言文件落位、SQLite 状态、异常路由。
- **非破坏验收**：保留 `phase_f_dry_run` 风格的 CLI，`no_files_moved=true`，先在临时库上跑全链路再碰真实库。
- **前端**：`vitest` 覆盖 viewModel 纯函数（筛选 / 拼贴 / 进度格式化等）；组件交互用测试库。
- **不可在 agent 会话里启动 Tauri GUI / WebView2**（沿用 `AGENTS.md` 硬约束）；前端验证用 `npm run build` + `tsc --noEmit` + HTTP 探测，可视检查由用户在自己的浏览器做。

## 12. 复用 vs 砍掉

### 12.1 复用（已验证、踩过坑的核心）

- `identifier` — 番号正则识别。
- `scanner` — 识别 / ffprobe 媒体信息 / 采样指纹去重。
- `nfo` — quick-xml NFO 解析（含写回标准 NFO）。
- `storage` — SQLite 仓储（works 32 列 + 关系表 + file_versions）。
- `resource_pool.rs` — 统一资源池扫描（按类型 + 番号聚合），支撑"资源目录"单一配置。
- archive 移动规则 / 边界校验 / 冲突安全后缀。
- `migration.rs` 的迁移编排（多版本合并、自包含目录）。

### 12.2 砍掉 / 重写

- ingest-first 的逐条 review / resolve / merge 重人工模型 → 换成"默认全过 + 极小异常队列"。
- `commands.rs` 的命令堆砌 → 重新组织成 daemon 控制接口。
- 超大单体 `App.tsx` → 按六大页面拆分组件，全新深色沉浸 UI。
- 现有"源根 + 外部图片库"分离配置 → 合并为单一"资源目录"。

## 13. 迁移与风险

- **数据迁移**：现有 `library.sqlite` 的 works / 关系表 / file_versions 保留；新增表与字段通过 migration 脚本补齐；用户字段（rating / watch_status / progress）旧数据为空，后续随使用填充。
- **刮削源稳定性**：第三方站点可能改版 / 需代理 / 封禁。多源回退 + 手动兜底 + `scrape_jobs` 重试缓解；刮削源适配器隔离，便于单独修。
- **aria2 集成复杂度**：BT 多文件完成判定、未配置 aria2 时的启发式兜底需仔细处理边界；先 RPC 轮询，回调与启发式分阶段加。
- **内置播放器**：本期只做基础播放 + 进度记录；视频解码依赖系统/WebView，大文件 / 高码率体验需实测。

## 14. 未决 / 后续

- 内置播放器的具体技术选型（WebView `<video>` vs 系统播放器封装）在实施期定。
- lightbox、文件对比、去搜索结果等弹窗按 §7.3 规范落地。
- 非日厂 / 短视频的后续支持（本期搁置区暂存）。
- 感知 / 近似去重（本期仅精确指纹去重）。
- 收藏夹智能夹（按条件自动归集）。
- **aria2 下载集成**：本期仅对接 aria2 的"下载完成"信号来触发入库；后续把下载任务管理（添加任务 / 暂停 / 限速 / 队列）也集成进 app，成为一站式"下载即归档"。
- **NAS 式远程 / 移动端浏览**：后续让后台服务对外暴露 HTTP，移动端 / 远程浏览器可自由浏览已入库作品（鉴权 + 可选转码 + 外网访问）。本期"daemon + 本地控制接口 + SQLite"的服务端化架构已为此预留演进路径——前端是独立的客户端，后续可替换 / 新增移动端与远程 web 客户端，复用同一套服务与数据。
